## 0. What the project is

**Goal:** Does providing LLM-generated seed inputs to a fuzzer improve code coverage
compared to empty or random-byte baselines?

**Why it matters:**
- Fuzzers rely on their initial corpus to bootstrap coverage. A better starting corpus
  means faster coverage growth and more bugs found per CPU-hour.
- Prior LLM-for-fuzzing work mostly measures *seed-time* coverage (how much the seeds
  cover before the fuzzer runs). The harder, more meaningful question is *campaign-time*
  coverage: do the seeds help the fuzzer discover more code over a real fuzzing budget?
- An LLM that generates structurally valid inputs can potentially reach code paths that
  a pure mutation fuzzer would take hours or days to reach from random bytes.

**Research claim (end-of-quarter target):**
"An LLM with access to source code + coverage gaps produces a seed corpus that beats
random seeds on real fuzzing campaigns across multiple targets."

**Pipeline overview:**
```
1. Build coverage-instrumented target binary
2. Extract upstream tests, compute uncovered branches (coverage gaps)
3. LLM seed synthesis — 5 prompt variants
4. Measure seed-time coverage (M1: total edges, M2: hard-branch hit rate)
5. [Future] Real fuzzing campaigns using the seeds
```

## 1. Experiment design — the 5-variant ablation

Same 5-cell design used on both RE2 v2 and harfbuzz. All cells produce exactly
150 seeds per model (synthesis retries until full; subsampled deterministically to
exactly 150).

| Variant | Source code? | Tests? | Coverage gaps? | Label |
|---|:---:|:---:|:---:|---|
| v0_none        | ❌ | ❌ | ❌ | Blind (no context) |
| v1_src         | ✅ | ❌ | ❌ | Source only |
| v2_src_tests   | ✅ | ✅ | ❌ | Source + tests |
| v3_all         | ✅ | ✅ | ✅ | Full context |
| v4_src_gaps    | ✅ | ❌ | ✅ | Source + gaps (no tests) |

**Random baseline:** 150 seeds generated by format-aware random generation matching
the target input format (regex harness bytes for RE2, uniform random bytes for harfbuzz).

**Hard-branch M2 metric (same filter for both targets):**
A branch qualifies as "hard" only if it is hit by ≥1 structured smoke seed AND hit
by 0 random-format seeds. This ensures the random baseline always scores 0%,
making the metric meaningful. Targets are split ~2/3 shown (given to the LLM in the
prompt) / ~1/3 held-back (not shown, tests generalization).

## 2. Target 1 — RE2 (regex engine, text format) — v2 design

RE2 is a C++ regex library. Seeds are binary-format inputs where the first 2 bytes
are flag bits and the remainder is an ASCII regex string. The LLM generates regex
syntax directly.

**Why RE2:** Text format — the LLM can reason about regex syntax at the character
level and construct patterns that exercise specific parser branches.

**RE2 v2 design notes:**
- Hard-branch filter: rand_hits==0 AND struct_hits≥1 (identical to harfbuzz).
  An earlier RE2 experiment (ablation_v3) used a weaker filter; random bytes hit
  41/50 (82%) of those "hard" targets, invalidating the M2 metric entirely.
- RE2 v2 found only **15 hard branches** (10 shown / 5 held-back) across
  parse.cc, regexp.cc, simplify.cc, and walker-inl.h. The low count itself is a
  finding: text-format RE2 has far less LLM-exclusive territory than binary-format
  harfbuzz (15 vs 58 qualifying branches), because random regex strings accidentally
  cover most RE2 parsing paths.
- Model: llama-3.1-8b-instruct only. Exactly 150 seeds per cell.

### RE2 v2 — M1 (total edges covered at seed time, out of 3380 total)

| cell | edges covered | % of total | Δ vs random |
|---|---:|---:|---:|
| random       | 1160 | 34.3% | baseline |
| v0_none      | 1202 | 35.6% | +42 (+3.6%) |
| v1_src       | 1346 | 39.8% | +186 (+16%) |
| v2_src_tests | **1376** | **40.7%** | **+216 (+19%)** |
| v3_all       | 1249 | 37.0% | +89 (+7.7%) |
| v4_src_gaps  | 1262 | 37.3% | +102 (+8.8%) |

**M1 findings:**
- Source context gives a clear M1 boost: v1 and v2 gain +16–19% over random.
- v2 (source + tests) is the best cell — test examples improve edge coverage beyond
  source alone.
- v3 and v4 (with gap context) score below v1/v2. Adding gap descriptions may
  focus the LLM on a narrow set of branches at the cost of diversity.
- v0 (no context) barely beats random (+3.6%), confirming context is the driver.

### RE2 v2 — M2 (hard-branch hit rate, 15 targets, rand_hits==0 filter)

| cell | all 15 | shown 10 | held-back 5 |
|---|---|---|---|
| random       | 0/15 (0%)  | 0/10 (0%)  | 0/5 (0%)   |
| v0_none      | 5/15 (33%) | 3/10 (30%) | 2/5 (40%)  |
| v1_src       | 8/15 (53%) | 4/10 (40%) | 4/5 (80%)  |
| v2_src_tests | 8/15 (53%) | 4/10 (40%) | 4/5 (80%)  |
| v3_all       | **9/15 (60%)** | **5/10 (50%)** | 4/5 (80%) |
| v4_src_gaps  | 8/15 (53%) | 4/10 (40%) | 4/5 (80%)  |

**M2 findings:**
- Random correctly scores 0% — the hard-branch filter is working.
- Strong LLM signal: even v0 (blind) hits 33%; best cell (v3_all) hits 60%.
- Source context is the critical ingredient: v0→v1 jumps from 33% to 53%.
- Adding tests or gaps beyond source gives marginal additional gain (+0–7%).
- Held-back hit rates are high (80% for v1–v4) — the LLM generalizes to
  branches it was not explicitly shown, suggesting it is learning broader
  parsing structure, not just memorizing prompted branches.

## 3. Target 2 — harfbuzz (font shaping engine, binary format)

Harfbuzz is a production C++ OpenType/TrueType font shaping library used in
browsers, operating systems, and text renderers. The fuzzer harness feeds raw
binary font file bytes directly to `hb_blob_create → hb_face_create →
hb_font_create → hb_shape`.

**Why harfbuzz:**
- Binary input format — the LLM cannot craft valid inputs character by character.
  It must know the TrueType/OpenType binary wire format (sfnt header, table
  directory, GDEF/GSUB/GPOS/cmap/head/hhea/maxp/loca/glyf, CFF, variable-font
  tables, etc.) to target specific code paths.
- Tests whether LLM coverage benefit is format-specific (text: yes) or general.
- Widely deployed real-world software with deep, complex parsing.

### Hard-branch selection for harfbuzz

1. Built 68 minimal TTF/OTF skeleton structures covering 15 distinct font table
   groups (GSUB with script tags, GPOS, GDEF, cmap formats 4/12/13, variable
   fonts fvar/avar/gvar, color fonts COLR/CPAL, CFF, MATH, AAT morx/feat,
   loca+glyf, etc.)
2. Generated 100 random binary blobs as the "random corpus"
3. Hard branches = hit by ≥1 skeleton, hit by 0 random blobs
4. Found **58 hard branches** across 17 harfbuzz source files
5. Split: 30 shown (given to LLM in prompt) / 20 held-back (not shown)

The 58 qualifying branches vs 15 for RE2 (at rand_hits==0) is itself a finding:
binary font parsing code has far more LLM-exclusive territory than text regex parsing.

### Harfbuzz synthesis prompt

LLM is asked to produce base64-encoded font file bytes with a 3-step reasoning trace:
(1) which branch am I targeting and which side (TRUE/FALSE), (2) what OpenType table
structure triggers that code path, (3) what specific bytes force that branch side.
3 inputs per API call, 8192 output token budget (4096 was too small — the model
generates full well-formed OTF fonts ~16KB+).

### Harfbuzz — llama-3.1-8b results (complete, 150 seeds/cell)

#### M1 — total edges covered at seed time (out of 4628 total)

| cell | edges covered | % of total | Δ vs random |
|---|---:|---:|---:|
| random       | 554 | 12.0% | baseline |
| v0_none      | 564 | 12.2% | +10 (+1.8%) |
| v1_src       | 551 | 11.9% | -3 (-0.5%) |
| v2_src_tests | 560 | 12.1% | +6 (+1.1%) |
| v3_all       | 563 | 12.2% | +9 (+1.6%) |
| v4_src_gaps  | 558 | 12.1% | +4 (+0.7%) |

M1 differences are within ±2% of random. On harfbuzz, llama provides no
meaningful seed-time total-coverage advantage over random bytes.

#### M2 — hard-branch hit rate (50 targets, rand_hits==0 filter)

| cell | all 50 branches | shown 30 | held-back 20 |
|---|---|---|---|
| random       | 0/50 (0%)  | 0/30 (0%)  | 0/20 (0%)   |
| v0_none      | 2/50 (4%)  | 0/30 (0%)  | 2/20 (10%)  |
| v1_src       | 0/50 (0%)  | 0/30 (0%)  | 0/20 (0%)   |
| v2_src_tests | 1/50 (2%)  | 0/30 (0%)  | 1/20 (5%)   |
| v3_all       | 2/50 (4%)  | 0/30 (0%)  | 2/20 (10%)  |
| v4_src_gaps  | 1/50 (2%)  | 0/30 (0%)  | 1/20 (5%)   |

**Key findings:**
- Random correctly scores 0% — metric is valid.
- LLM barely clears zero: best cell hits only 2/50 (4%), the same as v0 (no context).
- All shown-branch scores are 0% across every cell — the LLM cannot translate a
  text description of a specific font branch into bytes that hit that exact branch.
- No context-scaling signal: adding source, tests, and gaps does not improve M2.
- The held-back anomaly mirrors the shown=0 result: incidental hits by seeds that
  stumble into nearby parsing paths, not directed targeting.

## 4. Head-to-head: RE2 v2 vs harfbuzz (both llama, same methodology)

This is the primary comparison. Both experiments use the identical framework:
same 5 variants, same 150-seed normalization, same rand_hits==0 hard-branch filter,
same measurement scripts.

### M1 comparison

| target | best LLM cell | best edges | Δ vs random | random edges |
|---|---|---:|---:|---:|
| RE2 v2    | v2_src_tests | 1376/3380 (40.7%) | +216 (+19%) | 1160 (34.3%) |
| harfbuzz  | v0_none / v3_all | 564/4628 (12.2%) | +10 (+2%) | 554 (12.0%) |

RE2 source-context cells beat random by ~19%; harfbuzz cells are essentially flat.

### M2 comparison

| target | hard branches found | random M2 | best LLM M2 | LLM lift |
|---|---:|---:|---:|---:|
| RE2 v2    | 15 (rand_hits==0) | 0%  | 60% (v3_all) | +60 pp |
| harfbuzz  | 58 (rand_hits==0) | 0%  | 4% (v3_all)  | +4 pp  |

On RE2, the LLM reaches 60% of hard branches — strong signal. On harfbuzz, 4% —
near-zero. Both use the same valid metric (random=0%).

### What explains the difference

| factor | RE2 | harfbuzz |
|---|---|---|
| Input format | text (ASCII regex) | binary (font bytes) |
| LLM can construct inputs precisely? | yes — character by character | no — requires byte-level binary format knowledge |
| Hard branches in LLM-exclusive territory | 15 | 58 |
| Best M2 (llama) | 60% | 4% |
| M1 benefit from source context | +19% | +2% |
| Context scaling (v0→v3)? | yes, M2 improves monotonically | no signal |

**The central finding:** LLM-guided coverage gains are strongly format-dependent.
When the LLM can reason precisely about input structure (regex text), it produces
significantly better seeds on both M1 and M2. When the input is an opaque binary
format, gains disappear — at least for llama-3.1-8b.

The RE2 result also resolves a methodology concern: the low harfbuzz numbers are not
due to a broken metric or instrument — the same metric correctly shows large LLM
advantage on RE2.

## 6. Design decisions and known bugs fixed

**RE2 v2 vs old RE2 ablation_v3 — what changed:**
1. **Hard-branch filter strengthened:** ablation_v3 used a weaker filter; random hit
   41/50 (82%) of those targets. v2 requires rand_hits==0. Random now scores 0%.
2. **Seed count normalized:** ablation_v3 had llama at 83–103 seeds vs 100 for Claude —
   a known confound. v2 normalizes to exactly 150 per cell for all models.
3. **Only 15 hard branches found:** at the rand_hits==0 threshold, RE2 has very few
   LLM-exclusive branches. This is a real finding, not a bug. The low count makes
   statistical power weak but the signal is still clear (0% random vs 60% best LLM).

**Harfbuzz-specific decisions:**
- Base64-encoded font blobs in JSON response, 3 inputs per API call, 8192 token output
  budget (needed for full OTF fonts the model generates).
- Prompt includes a 3-step reasoning chain: which branch, what table structure, what bytes.
- 30 shown / 20 held-back split on 58 hard branches.

**Cache collision bug (fixed):** On restart after credit exhaustion, synthesis replayed
cached API responses, freezing the seed count. Fixed via `--attempt-offset N` CLI flag
that shifts the `run_id` cache key component, guaranteeing misses across restarts.

**UnicodeEncodeError fix:** Llama occasionally generates regex patterns containing
Unicode surrogate characters (\\ud800). The seed ID hash function crashed on these.
Fixed by encoding with `errors="replace"`.

## 8. Hypotheses — prediction vs result

| Hypothesis | Prediction | RE2 v2 (llama) | harfbuzz (llama) |
|---|---|---|---|
| H-A: LLM beats random on M1 | ≥+10% edge advantage | +19% (v2_src_tests). **Supported.** | +2% max. **Not supported.** |
| H-B: v3/v4 hit ≥20% hard branches (M2) | union_frac ≥ 0.20 | 60% (v3_all). **Strongly supported.** | 4% max. **Not supported.** |
| H-C: Shown > held-back on M2 | shown frac > held_back | Shown=50%, held=80%. **Reversed — held higher.** | Shown=0%, held>0%. **Reversed.** |
| H-D: Binary harder than text for LLM | harfbuzz gain < RE2 gain | RE2 M1 +19%, M2 60% | harfbuzz M1 +2%, M2 4%. **Strongly supported.** |
| H-E: Hard-branch filter prevents random inflation | random M2 ≈ 0% | 0%. **Confirmed.** | 0%. **Confirmed.** |

H-C (shown > held-back) is consistently reversed on both targets. On RE2 this is
less concerning because the held-back score is high (80%), not just unexpectedly low for shown.
On harfbuzz, shown=0% across every cell suggests llama cannot use the branch descriptions
at all — the shown/held distinction is meaningless at near-zero M2.

## 9. Open questions for reviewer

1. **Is H-D the headline result?** "LLMs help for text-format targets but not
   binary-format targets (at least for small open-weights models)" — is this a crisp,
   publishable claim, or does it need frontier model confirmation first?

2. **What will Claude Sonnet show on harfbuzz?** If Sonnet M2 is also ~4%, the finding
   is model-size-independent and the format argument is solid. If Sonnet shows 20%+,
   the issue is llama's capability, not the binary format per se.

3. **The shown < held-back anomaly** — consistent across both targets. On RE2 the held
   rate (80%) is genuinely high, so the LLM is generalizing well but its prompt-targeted
   performance lags. Why? Possible: gap descriptions focus generation on narrow patterns
   that miss nearby branches, while unconstrained generation explores more broadly.

4. **RE2 v2 statistical power:** Only 15 hard branches, 10 shown / 5 held-back. The
   80% held-back rate (4/5) rests on 5 data points. Confidence intervals are wide.
   Expanding the structured smoke corpus to find more qualifying branches would help.

5. **Should we run harfbuzz campaigns?** We have no campaign-time data for harfbuzz.
   A 1-hour campaign with Claude Sonnet seeds (even at low M2) could test whether
   structurally valid fonts give the fuzzer better starting points for mutation.

6. **Is 150 seeds sufficient?** The 95% CI on a 4% hit rate with n=150 is [0%, 8%].
   Bumping to n=500 per cell (~$15 total for Claude) would substantially narrow CIs.
