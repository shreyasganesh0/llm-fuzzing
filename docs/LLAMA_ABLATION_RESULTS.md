## 0. Project goal

Does providing LLM-generated seed inputs improve code coverage compared to random
or empty baselines?

**Pipeline:**
1. Build a coverage-instrumented binary for the target library.
2. Compute "hard" uncovered branches — branches reachable by at least one structured
   smoke seed but hit by **zero** random-format seeds. This guarantees the random
   baseline always scores 0% on M2, making the metric meaningful.
3. Run 5 prompt variants × 1 model × 150 seeds per cell (synthesis retries until
   exactly 150 seeds; deterministically subsampled to exactly 150 before measurement).
4. Measure two metrics at seed-time (before any fuzzing campaign):
   - **M1** — total unique edges covered by the 150-seed corpus union.
   - **M2** — fraction of the pre-selected hard target branches hit by the corpus union.

**Model tested:** llama-3.1-8b-instruct (served via UF AI API, vLLM).

**Targets:** RE2 (C++ regex engine, text-format inputs) and harfbuzz (C++ OpenType
font shaping engine, binary-format inputs). Same 5-variant design on both.

---

## 1. The 5-variant ablation design

Same design on both targets. Variants differ in what context is included in the prompt.

| Variant | Source code? | Upstream tests? | Coverage gaps? | Short label |
|---|:---:|:---:|:---:|---|
| v0_none        | ❌ | ❌ | ❌ | Blind (no context) |
| v1_src         | ✅ | ❌ | ❌ | Source only |
| v2_src_tests   | ✅ | ✅ | ❌ | Source + tests |
| v3_all         | ✅ | ✅ | ✅ | Full context |
| v4_src_gaps    | ✅ | ❌ | ✅ | Source + gaps (no tests) |

**Random baseline:** 150 seeds generated by format-aware random generation (random
regex harness bytes for RE2; uniform random binary blobs for harfbuzz).

**Coverage gap targets are split:**
- RE2 v2: 15 hard branches total — 10 shown in prompt / 5 held-back.
- Harfbuzz: 50 hard branches total — 30 shown in prompt / 20 held-back.

Shown branches are given to the LLM in variants v3 and v4. Held-back branches test
whether the LLM generalises beyond what was explicitly requested.

## 2. RE2 results (regex engine, text-format inputs)

**Target:** google/re2 — a C++ library for safe, linear-time regular expression matching.
**Input format:** Binary blob where bytes 0–1 are flag bits and the remainder is an
ASCII regex string. The LLM generates the regex string directly.
**Hard branch count:** 15 qualifying branches found (10 shown / 5 held-back). The low
count is itself a finding: random regex strings accidentally cover most RE2 parsing
paths, leaving little LLM-exclusive territory.
**Total edges in binary:** 3,380.

### RE2 M1 — total edges covered (150-seed corpus union)

| Cell | Edges covered | % of 3380 | Δ vs random |
|---|---:|---:|---:|
| random (baseline) | 1,160 | 34.3% | — |
| v0_none           | 1,202 | 35.6% | +42  (+3.6%) |
| v1_src            | 1,346 | 39.8% | +186 (+16.0%) |
| v2_src_tests      | **1,376** | **40.7%** | **+216 (+18.6%)** |
| v3_all            | 1,249 | 37.0% | +89  (+7.7%) |
| v4_src_gaps       | 1,262 | 37.3% | +102 (+8.8%) |

**M1 observations:**
- Source context gives a clear boost: v1 and v2 are +16–19% over random.
- v2 (source + tests) is the top cell — test examples help more than explicit gap
  targeting for raw coverage breadth.
- v3 and v4 (gap-targeted) underperform v2, suggesting that narrow gap targeting
  trades breadth for precision.
- v0 (blind) is only marginally better than random (+3.6%), confirming the baseline
  sanity check.

### RE2 M2 — hard-branch hit rate (union fraction, 15 targets)

| Cell | All (15) | Shown (10) | Held-back (5) | % seeds with any hit |
|---|:---:|:---:|:---:|:---:|
| random (baseline) | 0/15 (0%) | 0/10 (0%) | 0/5 (0%) | 0.0% |
| v0_none           | 5/15 (33%) | 3/10 (30%) | 2/5 (40%) | 15.3% |
| v1_src            | 8/15 (53%) | 4/10 (40%) | 4/5 (80%) | 12.7% |
| v2_src_tests      | 8/15 (53%) | 4/10 (40%) | 4/5 (80%) | 13.3% |
| v3_all            | **9/15 (60%)** | 5/10 (50%) | **4/5 (80%)** | **18.0%** |
| v4_src_gaps       | 8/15 (53%) | 4/10 (40%) | 4/5 (80%) | 10.7% |

**M2 observations:**
- Random scores exactly 0% as intended (the hard-branch filter is working correctly).
- Even blind v0 hits 33% of hard branches — RE2 syntax knowledge is baked into the
  model's weights.
- All LLM variants beat random by a wide margin (33–60% vs 0%).
- v3_all (full context) is the top cell at 60% overall.
- Held-back performance is consistently ≥ shown performance (80% held-back vs 40–50%
  shown in v1/v2/v3/v4). This suggests the LLM generalises beyond the explicitly
  listed gaps.
- The gap-targeted variants (v3, v4) outperform the no-gap variants (v1, v2) on M2,
  confirming that gap hints redirect synthesis toward the hard branches.

## 3. Harfbuzz results (font shaping engine, binary-format inputs)

**Target:** harfbuzz — a C++ implementation of the OpenType font shaping engine used
in browsers, Android, and most Linux desktops.
**Input format:** Raw binary font file bytes fed directly to `hb_blob_create`. The
LLM must produce base64-encoded font-like binary blobs.
**Hard branch count:** 50 qualifying branches (30 shown / 20 held-back). More
LLM-exclusive territory than RE2 because random binary blobs rarely trigger
deep OpenType table parsing.
**Total edges in binary:** 4,628.

### Harfbuzz M1 — total edges covered (150-seed corpus union)

| Cell | Edges covered | % of 4628 | Δ vs random |
|---|---:|---:|---:|
| random (baseline) | 554 | 12.0% | — |
| v0_none           | 564 | 12.2% | +10 (+1.8%) |
| v1_src            | 551 | 11.9% | −3  (−0.5%) |
| v2_src_tests      | 560 | 12.1% | +6  (+1.1%) |
| v3_all            | 563 | 12.2% | +9  (+1.6%) |
| v4_src_gaps       | 558 | 12.1% | +4  (+0.7%) |

**M1 observations:**
- All variants are within ±2% of the random baseline — effectively flat.
- Source context provides no meaningful M1 boost for binary-format inputs.
- Interpretation: 12% of harfbuzz edges are reachable by any roughly-valid binary
  blob; the remaining 88% require deeply structured font data that llama-3.1-8b
  does not produce reliably.

### Harfbuzz M2 — hard-branch hit rate (union fraction, 50 targets)

| Cell | All (50) | Shown (30) | Held-back (20) | % seeds with any hit |
|---|:---:|:---:|:---:|:---:|
| random (baseline) | 0/50 (0%)  | 0/30 (0%)  | 0/20 (0%)   | 0.0% |
| v0_none           | 2/50 (4%)  | 0/30 (0%)  | 2/20 (10%)  | 1.3% |
| v1_src            | 0/50 (0%)  | 0/30 (0%)  | 0/20 (0%)   | 0.0% |
| v2_src_tests      | 1/50 (2%)  | 0/30 (0%)  | 1/20 (5%)   | 0.7% |
| v3_all            | 2/50 (4%)  | 0/30 (0%)  | 2/20 (10%)  | 0.7% |
| v4_src_gaps       | 1/50 (2%)  | 0/30 (0%)  | 1/20 (5%)   | 0.7% |

**M2 observations:**
- All variants score ≤4% on all-50 — near-zero signal for llama on binary format.
- Zero shown branches hit in any variant. The model cannot construct font bytes that
  target specific parser branches even when those branches are described in the prompt.
- The 1–2 hits that appear are exclusively in the held-back set — likely coincidental
  coverage from partially-valid blobs, not intentional targeting.
- Prompt context (source, tests, gaps) makes no detectable difference for a small model
  on a binary-format target.

## 4. Head-to-head: RE2 vs harfbuzz (same model, same design)

| Metric | RE2 (text format) | Harfbuzz (binary format) |
|---|---|---|
| Hard branches qualifying | 15 (few) | 50 (many) |
| Random M2 | 0% ✅ | 0% ✅ |
| Best LLM M2 (all targets) | **60%** (v3_all) | **4%** (v0/v3) |
| Best LLM M1 gain vs random | **+19%** (v2_src_tests) | **+2%** (v0_none) |
| Top M1 variant | v2_src_tests (source + tests) | v0_none (no context) |
| Top M2 variant | v3_all (full context) | v0_none / v3_all (tied, near-zero) |
| Context helps M1? | Yes, strongly | No |
| Context helps M2? | Yes, gap hints redirect synthesis | No detectable effect |

**Key finding:** Performance is strongly format-dependent. For RE2 (text), the LLM
produces syntactically valid regex strings that exercise real parser logic — gap
context makes this even more targeted. For harfbuzz (binary), the 8B parameter
llama model cannot reliably construct valid font binary structures, so all variants
collapse to near-random performance regardless of prompt content.

**Implication:** LLM seed synthesis is most valuable when (a) the input format is
textual or the LLM has strong prior knowledge of the binary format, and (b) the
target has code paths that are reachable by syntactically-correct inputs but not
by random bytes. The RE2 result is promising; harfbuzz requires either a larger or
more capable model, or a different synthesis approach (e.g., grammar-guided).

## 6. Methodology notes

- **Hard-branch filter (same for both targets):** A branch qualifies only if hit by
  ≥1 structured smoke seed AND by 0 random-format seeds. This makes the random
  baseline score exactly 0% on M2, ensuring the metric has a clean baseline.
- **Seed normalisation:** Synthesis retries until exactly 150 seeds are produced;
  surplus seeds are deterministically subsampled (RNG seed = 42) to exactly 150
  before measurement. This eliminates seed-count as a confound.
- **RE2 v2 design note:** An earlier RE2 experiment (ablation_v3, 100 seeds) used a
  weaker M2 filter where random bytes hit 41/50 (82%) of "hard" targets — that metric
  was meaningless. RE2 v2 applies the strict rand_hits==0 filter and uses 150 seeds.
  **Do not compare RE2 ablation_v3 numbers with RE2 v2 or harfbuzz.**
- **M2 union metric:** The reported fraction is the union across all 150 seeds — i.e.,
  how many distinct hard branches were hit by at least one seed in the corpus. This is
  the most optimistic (corpus-level) view of coverage.
- **M1 source roots:** Coverage binary has DWARF paths baked as
  `phase1_dataset/targets/src/<target>/upstream/...` (pre-rearchitecture prefix).
  The `--source-roots` flag is set to match; path mismatches silently return 0 edges.
